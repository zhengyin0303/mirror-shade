## 一、化妆知识速成(产品决策必备)

一套完整妆容的标准流程分五步,每步对应的产品和作用如下。这是美妆行业的通用框架,也是我们功能扩展的地图:

### 1. 底妆(打底,占化妆时间的一半)
| 产品 | 作用 |
|------|------|
| 妆前乳/隔离 | 平滑肤质、控油保湿,帮后续上妆服帖 |
| **粉底**(液/霜/气垫) | 统一肤色、遮盖瑕疵——**选错色号是最大痛点**(脸黑脖子白) |
| 遮瑕 | 局部加强遮盖黑眼圈、痘印 |
| 散粉/定妆喷雾 | 固定妆容、控制反光 |

### 2. 眉妆
| 产品 | 作用 |
|------|------|
| 眉笔/眉粉/染眉膏 | 填充眉毛稀疏处、修整眉型、调整眉色 |

眉型本身是强需求:标准眉、柔雾眉、野生眉、欧式挑眉等,**眉型换一换整张脸气质就变**,和我们的"唇型编辑"逻辑同构。

### 3. 眼妆(技术含量最高的部分)
| 产品 | 作用 |
|------|------|
| 眼影 | 眼窝晕染色彩,单色到多色渐变,有大量"盘"和配色玩法 |
| 眼线(笔/液) | 勾勒眼型,拉长/放大眼睛 |
| 睫毛膏/假睫毛 | 浓密纤长睫毛 |
| 美瞳 | 改变瞳色与放大瞳孔 |

### 4. 颊妆
| 产品 | 作用 |
|------|------|
| 腮红 | 脸颊血色感,位置和形状决定显嫩还是显成熟 |
| 修容 | 在脸缘/鼻侧打阴影,视觉收窄脸型 |
| 高光 | 颧骨/鼻梁/眉骨提亮,增加立体感 |

### 5. 唇妆(已完成 ✅)
口红/唇釉/唇线笔——我们的原型已覆盖颜色、质地、唇线、浓度。

## 二、各品类 AR 可实现性评估

技术底座不变:MediaPipe 468 关键点对**每个面部区域都有现成点位**(眉、眼、瞳、脸颊轮廓全有),所以"识别"都不是问题,差异在"渲染得像不像"和"与产品定位是否冲突"。

| 品类 | 识别难度 | 渲染难度 | 效果价值 | 定位风险 | 建议优先级 |
|------|---------|---------|---------|---------|-----------|
| 腮红 | 低(脸颊点位现成) | 低(柔边色斑,同口红配方) | 高,妆效立竿见影 | 无 | **P1** |
| 眉妆 | 低(每侧5个眉点) | 中(加深填充+眉型模板形变) | 高,换眉型=换气质 | 无 | **P1** |
| 眼影 | 低(眼周点位现成) | 中(眼窝区域渐变晕染,闭眼/眨眼要跟随) | 高 | 无 | P2 |
| 眼线 | 低 | 中(沿睑缘细线,精度要求高) | 中 | 无 | P2 |
| 睫毛 | 低 | 高(毛发级细节,贴图易假) | 中 | 无 | P3 |
| 修容/高光 | 低 | 中(脸缘阴影+提亮) | 中 | 低 | P3 |
| 粉底 | 低 | 低(全脸色彩映射) | 高(选色号刚需) | **高,见下** | P2(限定做法) |
| 美瞳 | 低(虹膜点位现成) | 中(瞳孔贴图+遮挡处理) | 中 | 无 | P3 |

**粉底的定位红线(重要决策)**:我们的立身之本是"诚实的镜子,不磨皮不滤镜"。粉底功能必须做成"**色号匹配工具**"——回答"哪个色号和我脖子的颜色最接近、不假白"——而**不做**磨皮遮瑕美化。前者是帮用户买对粉底(刚需+导购价值),后者就堕落成普通美颜相机了。这条红线建议写进产品原则。

## 三、口红衣橱(P0,二期第一个功能)

### 3.1 用户故事
"我已经有十几支口红,买新色号前想知道:这支和我已有的像不像?我现有的口红里哪支最配今天的妆?"

### 3.2 功能设计
1. **我的衣橱**:新增页签或入口,网格展示已录入的口红(色块+名称+质地),点击任意一支直接上唇试色;
2. **录入方式 A·从色板收藏**:试色时长按色号 → "加入我的衣橱";
3. **录入方式 B·自定义录入**:手动输入名称 + 选色(色轮/色卡)+ 质地,支持备注品牌色号(为将来接入品牌库预留字段);
4. **录入方式 C·拍照取色**(二期后段):拍口红膏体或手臂试色照,从照片取色入库;
5. **撞色预警(核心差异化)**:试任意新色时,系统自动与衣橱全量比对,若存在高相似色,顶部提示"⚠ 与你的「XX」相似度 92%"——买前查重,帮用户省钱;
6. **衣橱管理**:删除、重命名、按色系排序。

### 3.3 技术方案
- **相似度算法**:颜色换算到 Lab 空间(原型已有 rgb2lab),计算 ΔE 色差。ΔE<5 视为"几乎相同"(相似度>90%),ΔE 5–12 为"相近",映射成百分比展示。质地不同(哑光vs水光)时相似度上限打折(同色不同质地不算完全撞色);
- **存储**:浏览器 localStorage(站点独立、持久、无需账号),数据结构沿用项目方案预留的"产品"模型:`{id, name, brand?, shadeCode?, color, texture, addedAt}`;提供导出/导入(JSON 文本)防换机丢失;原生 App 阶段迁移到本地数据库+云同步;
- **UI**:面板新增"衣橱"页签,与一键试色/亲手上妆/唇型并列。

### 3.4 验收要点
录入→试涂→撞色提示链路完整;ΔE 阈值用真实口红色号对验证(如两支相近豆沙色应报警,豆沙vs正红不应);localStorage 持久性(关闭重开数据在);导出导入往返无损。

## 四、眉妆与腮红方案简述(P1,衣橱之后)

**腮红**:复用口红整套渲染管线(蒙版+color/multiply混合+浓度),蒙版改为脸颊两侧椭圆柔边区域(颧骨点位定位),提供 4-6 个腮红色 + 三种打法位置(苹果肌/斜扫/晒伤腮红)。预计 1-2 轮迭代。

**眉妆**:眉部各 5 关键点生成眉区蒙版;渲染 = 在现有眉毛上加深/染色(multiply 通道,自然保留真实眉流);眉型模板(标准/柔雾/野生/挑眉)= 蒙版形状的参数化变形,复用唇型控制点的技术思路。眉色联动发色推荐。预计 2-3 轮迭代。

两者完成后,"唇+眉+颊"已构成一个轻全妆,产品故事从"试口红"升级为"试妆容"。

## 五、需要准备的"材料"清单

| 材料 | 用途 | 获取方式 | 紧急度 |
|------|------|---------|--------|
| 通用色彩数据(口红/腮红/眼影色值) | 各品类色板 | 我直接定义(经验色值),无需采购 | 已具备 |
| 品牌色号数据(名称/系列/色值) | 真实色号库与衣橱录入 | 起步阶段人工整理热门色号(品牌官网/电商页),量大后再谈数据合作 | 衣橱C方式前 |
| 眉型模板 | 眉妆形变 | 我用参数曲线生成,无需美术素材 | P1前 |
| 多肤色测试样本 | 渲染参数标定 | 你找不同肤色唇色的朋友实测 | 持续 |
| 真实口红若干 | 对照渲染真实度、验证撞色阈值 | 你或家人现有的口红即可,无需新买 | 衣橱验收时 |

结论:**二期不需要采购任何东西**,唯一的"硬材料"是品牌色号数据,起步用人工整理完全够。

## 七、风险与原则

1. **范围纪律延续**:每个版本只做表内内容,做完验收再开下一个;
2. **诚实定位红线**:任何品类都不引入磨皮/瘦脸/大眼;粉底只做色号匹配;
3. **单文件架构的天花板**:功能到 v2.4 前后,代码量和复杂度会逼近单 HTML 文件的维护极限,届时迁移原生是工程必然,不是可选项;
4. **隐私延续**:衣橱数据同样只存本机,导出由用户主动操作。
